# i18n Frontend — Design Spec

## Goal

Make the frontend respect the user's language preference (pt-BR or en-US) already stored in the backend. All static UI text (labels, buttons, placeholders, messages) must render in the selected language.

## Approach

Use `next-intl` in **non-routing mode** (no `/en/`, `/pt-BR/` URL segments). The locale is driven by a cookie synced with the backend setting.

## Architecture

```
frontend/
  messages/
    en.json              # English translations (all namespaces)
    pt-BR.json           # Portuguese translations (all namespaces)
  src/
    i18n/
      request.ts         # Server-side: reads "locale" cookie, loads JSON
    app/
      layout.tsx         # NextIntlClientProvider wraps app (auto-inherits from request.ts)
  next.config.ts         # Wrapped with createNextIntlPlugin()
```

## Locale Resolution

1. `src/i18n/request.ts` reads cookie `locale` (fallback: `en`)
2. Loads `messages/{locale}.json`
3. `NextIntlClientProvider` in `layout.tsx` auto-inherits locale + messages
4. `<html lang>` set dynamically from locale

## Locale Switching Flow

1. User changes language in Settings page or Setup wizard
2. Backend is updated via PATCH `/api/v1/settings`
3. Cookie `locale` is set in the browser (`document.cookie`)
4. `router.refresh()` is called — server re-reads cookie, loads new messages
5. UI updates without full page reload

## Usage Pattern

```tsx
// In any 'use client' component:
import { useTranslations } from 'next-intl';

export default function DashboardPage() {
  const t = useTranslations('Dashboard');
  return <h1>{t('title')}</h1>;
}
```

Interpolation:
```json
{ "deviceCount": "{count} devices" }
```
```tsx
t('deviceCount', { count: 42 })
```

## Scope

### In scope
- Install `next-intl`
- Create `src/i18n/request.ts` config
- Wrap `layout.tsx` with `NextIntlClientProvider`
- Wrap `next.config.ts` with `createNextIntlPlugin()`
- Create `messages/en.json` and `messages/pt-BR.json` with all static strings
- Replace all hardcoded strings in 13 pages and ~10 components with `useTranslations()`
- Set cookie on language change in Settings page and Setup wizard
- Call `router.refresh()` after language change
- Dynamic `<html lang>` attribute
- On login/app load, sync cookie from backend settings if available

### Out of scope
- URL-based locale routing (no middleware, no `[locale]` segments)
- Dynamic data translation (ThingType names, Local names, API error messages)
- Additional languages beyond en and pt-BR
- Server-side rendering of translated content (all pages are 'use client')
- Date/number formatting (can be added later)

## Cookie Strategy

- Cookie name: `locale`
- Values: `en` or `pt-BR`
- Set via `document.cookie` (no httpOnly needed, client-readable)
- On Settings save or Setup complete: set cookie + `router.refresh()`
- On initial app load (AuthProvider or layout): if user is authenticated and settings are loaded, sync cookie from backend `language` field
